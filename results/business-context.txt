=== PROJECT CONCEPT ===

SmartLoop is a containerized edge service that provides closed-loop building automation for municipal buildings. It addresses the specific need identified by COP-PILOT Cluster 2: "What is still missing is the closed-loop automation layer that turns [IoT] insights into direct operational impact."

The service operates at two automation levels:

LEVEL 1 — Fully Automated (rule-based, no human intervention):
- Lighting control: turn off when occupancy = 0 for > 15 minutes
- Ventilation adjustment: reduce when CO2 < threshold AND occupancy = 0
- Schedule management: switch to night/weekend modes automatically
- Immediate energy savings (10-15%) from day one of deployment
- No ML training data required — works immediately

LEVEL 2 — Human-in-the-Loop (ML-driven recommendations):
- HVAC optimization: "Recommend reducing temperature by 2°C in zone B, saving €45/week" → facility manager approves/rejects
- Anomaly detection: "Energy consumption 30% above baseline, probable HVAC malfunction" → trigger maintenance ticket
- Schedule optimization: "Tomorrow forecast +35°C, recommend pre-cooling at 5:00" → approve/modify
- Occupancy prediction: learn weekly patterns, pre-condition spaces before arrival
- Trained on data collected during Level 1 operation (4-6 weeks sufficient)

=== TECHNICAL ARCHITECTURE ===

SmartLoop is a config-driven, containerized microservice that integrates with existing FIWARE-based IoT infrastructure:

Data Flow:
1. IoT sensors (already installed in Valencia municipal buildings) → FIWARE IoT Agents → Orion Context Broker (NGSI-LD)
2. SmartLoop subscribes to Context Broker for real-time sensor updates
3. Rule Engine (Level 1) and ML Engine (Level 2) process data
4. Decisions are published back to Context Broker as NGSI-LD entities
5. Actuator commands routed through IoT Agents to building systems
6. Facility Manager Dashboard shows status, recommendations, and analytics

Key Integration Points:
- FIWARE Orion Context Broker (NGSI-LD) — primary data interface (already deployed in CL2)
- FIWARE IoT Agents — sensor data ingestion (MQTT, HTTP, UL2.0)
- COP-PILOT ESO — service lifecycle management, SLA monitoring
- COP-PILOT SIF — secure zero-trust communication between components
- Prometheus/Grafana — performance metrics export
- Kubernetes — deployment target (K8s cluster available in UPV Campus domain)

Standards Compliance:
- NGSI-LD / NGSI-v2 for all data exchange with Context Broker
- FIWARE Smart Data Models for building entities (Building, BuildingSpace, Device, etc.)
- TMForum APIs for service catalog and ordering
- OAuth 2.0 / Keycloak for authentication
- MQTT for IoT device communication
- REST APIs for external integrations
- Docker/Helm for containerized deployment

Config-Driven Architecture (universal for any building):
The service is configured via a YAML configuration file per building:
- Building type (office, school, hospital, etc.)
- Floor area, number of zones
- HVAC system type (central, split, VRF)
- Available sensors and their NGSI-LD entity IDs
- Comfort parameters (temperature range, CO2 thresholds)
- Automation rules (customizable per building)
This enables deployment to ANY building with FIWARE-connected IoT sensors.

=== RISKS AND MITIGATIONS ===

1. Limited historical IoT data → Mitigation: Level 1 rules work without training data; ML models use transfer learning from open datasets (Building Data Genome 2.0, ASHRAE) + fine-tune on collected data
2. Actuator access in municipal buildings → Mitigation: Start with monitoring + recommendations (Level 2), add direct control progressively; COP-PILOT provides integration support for connected actuators
3. Ukraine war situation → Mitigation: All development is done remotely from secure locations; deployment is in Valencia; both ALD and ZNU have proven remote work capability and international travel logistics
4. On-site presence in Valencia → Mitigation: 3 planned trips (M2, M5, M7) with 5-7 day stays; remote access between trips via VPN to K8s cluster; weekly coordination calls with COP-PILOT
5. 8-month timeline → Mitigation: Two-level approach ensures deliverables at every stage; Level 1 is functional by M4 (intermediate report milestone)
6. FIWARE integration complexity → Mitigation: Leverage COP-PILOT technical mentoring programme and documentation; ALD has pre-prototyped NGSI-LD integration with local Orion CB instance
